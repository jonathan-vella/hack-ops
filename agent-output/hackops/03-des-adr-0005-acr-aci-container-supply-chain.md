## 🔍 Context

HackOps is moving to a containerized App Service deployment in
`swedencentral` with Azure SQL behind Private Endpoint and Private DNS.
This introduces two operational requirements from the architecture
assessment:

1. A secure container image supply chain for App Service runtime images.
2. A repeatable way to run SQL schema/seed operations from inside the
   VNet, because SQL public network access is disabled.

Direct schema execution from external CI runners is incompatible with the
private-only SQL posture. The platform needs an internal execution path
for migration/seed jobs while preserving managed identity and least
privilege controls.

## ✅ Decision

Adopt the following pattern:

- Use **Azure Container Registry (ACR) Standard** as the single image
  registry for HackOps runtime and migration images.
- Use **Azure Container Instances (ACI)** as an ephemeral,
  VNet-integrated execution target for SQL schema and seed operations.
- Keep App Service on managed identity pull (`AcrPull`) from ACR.
- Enforce image scanning (Trivy) in CI before push/deploy.

This decision maps to ADR-005 in
[02-architecture-assessment.md](./02-architecture-assessment.md).

## 🔄 Alternatives Considered

| Option                                                | Pros                                                                          | Cons                                                                                          | WAF Impact                             |
| ----------------------------------------------------- | ----------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- | -------------------------------------- |
| **ACR + ACI** (chosen)                                | Azure-native supply chain, VNet-integrated ephemeral execution, MI-based auth | Adds two resources and deployment complexity                                                  | Security: ✅, Operations: ✅, Cost: ➖ |
| GitHub Container Registry + external runner migration | Simple registry setup, no ACR resource                                        | Runner cannot reach private SQL without extra network plumbing; weaker Azure RBAC integration | Security: ⛔, Operations: ➖           |
| App Service startup migration logic                   | No extra compute service                                                      | Couples schema changes to app startup; risky rollouts and hard rollback semantics             | Reliability: ⛔, Operations: ⛔        |
| Azure Container Apps Jobs                             | Strong job semantics                                                          | Additional platform surface; less aligned with current App Service-first pattern              | Operations: ➖, Cost: ➖               |

## ⚖️ Consequences

### Positive

- Container artifacts are centralized in ACR with clear provenance.
- App Service pulls images via managed identity, avoiding registry secrets.
- SQL schema and seed tasks run from inside VNet using ephemeral ACI,
  satisfying private endpoint constraints.
- CI/CD becomes reproducible: build → scan → push → deploy → seed.
- Operational blast radius is reduced because ACI seed jobs are short-lived.

### Negative

- Additional Azure resources (ACR and ACI) increase baseline complexity.
- Deployment runbooks must include ACI lifecycle and failure handling.
- ACI cold-start and image-pull timing can increase release step duration.

### Neutral

- The pattern does not change the core app architecture (App Service + SQL);
  it adds a controlled operational path for deployment-time tasks.
- Cost impact remains modest for dev workloads because ACI is on-demand.

## 🏛️ WAF Pillar Analysis

| Pillar      | Impact | Notes                                                                                    |
| ----------- | ------ | ---------------------------------------------------------------------------------------- |
| Security    | ↑      | MI-based ACR pull, no registry credentials in app settings, private SQL access preserved |
| Reliability | →      | Core runtime unchanged; migration reliability improves through isolated seed jobs        |
| Performance | →      | Runtime request path unaffected; deployment phase may add image pull latency             |
| Cost        | ↓      | Slightly higher baseline (ACR Standard) plus on-demand ACI usage                         |
| Operations  | ↑      | Cleaner release workflow and repeatable migration/seed process                           |

## 🔒 Compliance Considerations

- **Private data path**: SQL remains private-only (`publicNetworkAccess: Disabled`).
- **Identity-first access**: App Service and seed process rely on managed identity/RBAC.
- **Credential minimization**: No long-lived DB or registry credentials in source control.
- **Supply-chain controls**: CI scanning before image promotion/deployment.
- **Auditability**: ACR push/pull and deployment actions are visible in Azure activity logs.

## 📝 Implementation Notes

- Bicep modules:
  - `modules/container-registry.bicep` for ACR Standard.
  - `modules/app-service.bicep` configured with
    `acrUseManagedIdentityCreds: true` and image tag parameterization.
  - SQL remains in `modules/sql-database.bicep` with PE + Private DNS + Entra-only auth.
- Deployment sequencing (from architecture handoff):
  - Phase 3: provision ACR + SQL
  - Phase 4: deploy App Service container
  - Phase 5: run ephemeral ACI schema/seed job inside VNet
- Rollback strategy:
  - Do not auto-run destructive migrations in app startup.
  - Keep schema scripts idempotent and versioned.
